The Challenge

One new stack.
Every week. For 12 weeks.

Each week you receive a new tech stack and a new problem. You have 7 days to build a working solution, document it, and demo it. The problems escalate in difficulty. By Week 12 you will be writing inline assembly to optimise a ZK cryptographic hash function. This is not a beginner hackathon.

You can use any preferred stack for the application layer — the weekly stack is the constraint for the core challenge. Fullstack means you own the entire system: backend, frontend, infrastructure, and everything in between.

⚡

Real-Time Systems

WebSockets, pub/sub, actor models, and distributed state. Build systems that handle thousands of concurrent users without dropping a message.

🔐

Privacy & Cryptography

ZK proofs, commitment schemes, nullifiers, and end-to-end encryption. Build systems where privacy is a mathematical guarantee.

🦀

Systems Programming

Rust, Zig, C++, and inline assembly. Write code that runs at the metal. No garbage collector. No safety net.

☁️

Cloud & Infrastructure

Terraform, Kubernetes, and multi-region deployments. Build infrastructure that survives region failures and scales automatically.

Timeline

12 weeks. 3 phases.

July 1 — September 21, 2026. Each week opens on Monday and closes on Sunday. Submit your PDF report, GitHub repo, and YouTube demo before the window closes.

Phase 1
The Modern Scale — Weeks 1–4
Focus: Speed, Type Safety, and Concurrent User Experiences
Week 1 · July 1–7
Go + Fiber + SvelteKit + Redis
Build a live bidding/auction engine with real-time price updates.
The Cook: Managing WebSocket state across multiple Go instances while keeping the Svelte UI reactively updated without lag.
Week 2 · July 8–14
Next.js + NestJS + PostgreSQL + Prisma
A multi-tenant SaaS platform with complex RBAC and dynamic schema generation.
The Cook: Handling deep-nested SQL relations and cache invalidation across a serverless frontend.
Week 3 · July 15–21
Flutter + Elixir/Phoenix + SQLite
A field-agent app that works in zero-connectivity zones and performs CRDT-based merge when back online.
The Cook: If two agents edit the same data offline, the merge must be perfect. No data loss. No conflicts.
Week 4 · July 22–28
C++ / Drogon + WebAssembly + React
A browser-based video/audio editor with heavy processing handled in WASM.
The Cook: Compiling C++ to WASM while maintaining a smooth React UI. Memory management is your problem now.
Phase 2
Privacy & Systems Architecture — Weeks 5–8
Focus: Security, Memory, and Native Performance
Week 5 · July 29 – Aug 4
Rust + Tauri + Vue.js
A peer-to-peer file sharing client with end-to-end local encryption.
The Cook: Bridging Rust memory safety with the Vue.js event loop without causing bottlenecks.
Week 6 · Aug 5–11
Haskell / IHP Framework + HTMX
A high-integrity financial ledger system with no JSON APIs.
The Cook: Using Haskell's type-safety to push HTML fragments directly. One type error in the DB kills the build.
Week 7 · Aug 12–18
Swift / Kotlin Native + Ktor
A low-latency biometric authentication system using native hardware.
The Cook: Managing native Camera/FaceID hardware and piping that data through a high-performance Kotlin backend.
Week 8 · Aug 19–25
Noir / Aztec + Rust + Next.js
A private voting platform. ZK proof generated client-side, verified on a Rust backend.
The Cook: Most will fail here due to proof-generation time and browser memory limits. Optimise or die.
Phase 3
The Broken & Hard Logic — Weeks 9–12
Focus: Low-level logic, Assembly, and Distributed Chaos
Week 9 · Aug 26 – Sep 1
Scala + Akka + Angular
A self-scaling social graph that re-shards its own database based on traffic patterns.
The Cook: The actor model in Scala. If one actor dies, does the whole system cascade?
Week 10 · Sep 2–8
Zig + Custom HTTP Server + Vanilla JS
Build a web server from scratch in Zig that serves a dashboard. No third-party libraries.
The Cook: No npm install. You write the headers, the buffers, and the TCP handling yourself.
Week 11 · Sep 9–15
Terraform + Kubernetes + Go + React
Deploy a globally distributed app across 3 regions with automated failover and 99.99% uptime.
The Cook: The judge kills one of your cloud regions. Your app must recover in under 5 seconds.
Week 12 · Sep 16–21
Inline Assembly + Rust + WASM + SolidJS
Take the ZK protocol from Week 8 and optimise the cryptographic hashing using inline assembly to make it 5x faster.
The Cook: One wrong move in assembly causes a total system crash. This is the final validation.

Judging Criteria

Scored like a project defence.

Every round is scored out of 100 by FLINN Software Labs judges. The standard is a Nigerian university final year project defence — technical depth, presentation, appearance, and a working demo all count.

🧠

Technical Depth (30pts)

Do you understand the system you built? Can you explain your architecture, your protocol choices, and your security assumptions under questioning?

⚙️

Implementation Quality (25pts)

Clean code, working tests, structured GitHub repo. If it doesn't run, you lose these points. Code quality is judged like a senior engineer would review it.

🎯

Innovation & Originality (20pts)

Did you go beyond the spec? Novel architecture, creative solutions, or unique application of the required stack will be rewarded.

🎤

Presentation & Defence (15pts)

Appearance, communication, and confidence matter. Present your work clearly. Defend it under questioning. This is a project defence, not a code review.

🖥️

Working Demo (10pts)

A live YouTube demo showing the system working end-to-end. A broken demo costs you all 10 points. Record it properly.

FAQ

Questions answered.

Do I need to know all 12 stacks before joining?

No. You learn as you go. Each week you have 7 days to pick up the stack and build with it. The resources section has curated docs for every week. The challenge is the learning curve — that's the point.

Is this only for ZK/cryptography developers?

No. Ghost Protocol is a fullstack hackathon. Only Weeks 8 and 12 involve ZK proofs. The other 10 weeks cover real-time systems, mobile, desktop, cloud infrastructure, and systems programming. Any serious fullstack developer can compete.

What do I submit each week?

Three things: a PDF report documenting your approach and architecture, a GitHub repository with your code, and a YouTube demo video showing the system working end-to-end. All three are required for a complete submission.

What happens if I miss a week?

You receive a score of zero for that round. Your cumulative score across all 12 rounds determines your final ranking. You can still submit in subsequent weeks — missing one round does not disqualify you.

How does the Team Code work?

The team leader registers and pays. A unique Team Code is generated on their dashboard. They share this code with teammates. Each teammate registers separately with the same role type, then enters the code on their dashboard to join the team.

Can I use any stack for the application layer?

Yes. The weekly stack constraint applies to the core challenge component. For supporting infrastructure, UI, or tooling, you can use whatever you are comfortable with. The judging focuses on the required stack implementation.

How are prizes paid out?

Prize money is paid via bank transfer within 7 days of the final round results being verified by FLINN Software Labs. Winners will be contacted directly.

What if my payment went through but my dashboard is locked?

Go to the pending payment page and click "Check Status". If still locked after a few minutes, contact FLINN Software Labs with your Paystack payment reference and we will resolve it.

Can students from any Nigerian university participate?

Yes. Ghost Protocol is open to all Nigerian university students and recent graduates regardless of institution.

Is the registration fee refundable?

No. Registration fees are non-refundable. Once payment is confirmed, your slot is secured for all 12 weeks.
